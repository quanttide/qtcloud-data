# 产品路线图

## 愿景

量潮数据云不是一个传统的 CRUD 数据平台，而是数据工程领域的**第二大脑**——一个承载规范、决策、处理逻辑和协作上下文的持久化知识系统。

核心命题：**规范数据工程信息在组织内部的流转，再在此基础上构建人、AI、机器的协作机制。**

## v0.1.0 — 信息模型地基

> 目标：定义第二大脑的核心信息类型及其关系，完成持久化与版本化。

- **Spec**：将数据工程规范（字段定义、质量规则、Schema）从静态文档变为版本化、可执行的一等公民，支持 AI 读写
- **Recipe**：将 `docs/spec/recipe.md` 中的概念实现为版本化的实体——封装 Blueprint + Schema + Processor + Handbook，支持执行追踪（每次执行结果、耗时、质量指标）
- **Decision**：记录数据工程中的关键决策——谁、何时、为何做出某个决策（如"为什么跳过某字段的清洗"），形成可追溯的决策链
- **Context（协作上下文）**：承载人→AI→机器的对话历史、意图、审批流，为后续协作机制奠基

关键交付：
- 知识库核心 API（非 CRUD，而是 Link / Trace / Evolve / Review）
- 版本管理：所有信息实体不可变，通过版本链演进（Event Sourcing 风格）
- 语义查询基础：支持按上下文搜索和关联追溯

## v0.2.0 — 规范闭环

> 目标：在真实项目中跑通"规范-实现-偏差-更新规范"的反馈闭环。

- AI 辅助创建 Recipe（从自然语言生成骨架）
- Recipe 执行引擎：自动化执行处理逻辑并记录结果
- 偏差检测：自动对比实际产出与 Spec 的偏差，生成 Decision 记录
- 人工 Review 面板：审批/修改 AI 提议的 Recipe 和 Spec 变更

## v0.3.0 — 人机协作

> 目标：构建人、AI、机器三者间的结构化协作机制。

- **Human → AI**：自然语言意图转 Recipe/Spec 草案
- **AI → Human**：执行结果分析、偏差预警、改进建议推送
- **Human Review → Machine Execution**：审批流驱动自动化执行
- **状态机/工作流引擎**：管理 Spec → Recipe → Execution → Review → Feedback 的全生命周期

## v0.4.0 — 组织的第二大脑

> 目标：成为团队共享的数据工程知识中枢。

- 语义搜索：基于 pgvector 的知识检索
- 溯源追踪：从产出物追溯到 Recipe、Spec、原始决策
- 团队协作：多用户上下文共享、通知、变更历史
- CLI 工具（`qtctl`）：作为 AI Agent 和自动化流水线的入口，输出结构化、机器可读

## v1.0.0 — 里程碑

- 完整的人-AI-机器协作闭环上线
- 至少在一个真实数据工程场景中验证"第二大脑"有效减少信息丢失和返工
- 开源核心模型与 API 规范

---

## 技术演进

### 阶段一（v0.1.0~v0.2.0）
- Provider（FastAPI）重构为知识库 API
- 数据库从 SQLite 迁移至 PostgreSQL + pgvector
- 引入事件溯源（Event Sourcing）存储所有信息变更
- CLI 基础框架搭建

### 阶段二（v0.3.0~v0.4.0）
- 引入工作流引擎
- CLI 作为 AI Agent 入口成熟
- Flutter Studio 退化为移动端查看器；主力交互移至 Web + CLI + AI API
