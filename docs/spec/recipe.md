# 数据工程方案(`Recipe`)

命名理由

选用 `Recipe` 作为处理能力交付物名称，因其精准传达了“标准化、可复用、声明式数据处理逻辑”的核心特征。该术语已在 DataKitchen、AWS Glue DataBrew 等主流 DataOps 平台中成为行业惯例，既能清晰区别于结果型资产（如 datasets/），又避免了“方案”“服务”等词的模糊性与营销色彩。同时，中文语境下可对应为“数据工程方案”，兼顾本地沟通习惯与技术严谨性，实现国际命名与本土表达的有机统一。

📜 正式定义

数据工程方案（Recipe） 是一个版本化的、自包含的标准化数据能力交付单元，用于在符合输入规范的前提下，可重复、可验证地将原始数据转化为可信输出。  
它封装了完整的处理意图、接口契约、执行逻辑与使用说明，不依赖外部上下文即可独立部署与运行，是数据工程能力的产品化载体。

🧩 核心组成

每个数据工程方案必须包含以下一等公民组件：
组件   作用   示例
Blueprint（蓝图）   描述业务目标与处理意图   “清洗多源问卷中的邮箱与日期字段”

Schema（规范）   定义输入/输出的数据结构与质量规则   input_schema.yaml, output_schema.yaml

Processor（处理器）   可执行的处理逻辑（代码、SQL、规则引擎等）   processor.py

Handbook（手册）   使用说明、依赖项、限制条件与示例   handbook.md

（推荐）Tests（测试）   自动化数据质量校验   tests/email_format_test.py

所有组件打包为单一 ZIP 文件，命名格式：{业务场景}_数据工程方案_v{主}.{次}.zip

🔑 关键特征

- ✅ 可复用：同一方案可应用于任意符合其输入规范的新数据集；  
- ✅ 不可变：发布后内容冻结，通过版本迭代演进；  
- ✅ 自描述：无需额外解释即可理解其用途与约束；  
- ✅ 可验证：支持自动化测试以确保输出一致性；  
- ✅ 与数据解耦：方案本身不含原始数据，仅定义“如何处理”。

📂 系统定位

/artifacts/
├── recipes/          ← 工程系统目录（英文，保持精确）
│   └── questionnaire_cleaning_v1.0.zip
└── datasets/
    └── questionnaire_20260116.parquet  ← 由该方案生成

- recipes/：存放所有数据工程方案（技术命名，国际惯例）  
- datasets/：存放方案执行结果（数据产出）

二者构成 “能力（How）” 与 “结果（What）” 的清晰分离。

⚠️ 重要澄清

本方案 不包含：
- 定制开发服务  
- 驻场实施  
- 运维支持  
- 原始数据  
它是一个纯资产交付物，客户需具备基础数据平台能力以部署和运行。

✅ 总结一句话

数据工程方案（Recipe） = 蓝图 + 规范 + 处理器 + 手册 的标准化、可复用、版本化聚合体，是数据工程能力的产品化最小单元。

此定义既尊重了你的架构设计，又对齐了 DataOps 行业实践，同时兼顾中文语境下的专业表达。
