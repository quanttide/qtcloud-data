## v0.1.0 — Blueprint 页面

核心目标：让 Studio 能够展示和操作数据蓝图，覆盖列表浏览和详情查看。

### 页面规划

| 路由 | 页面 | 说明 |
|------|------|------|
| `/blueprints` | 蓝图列表 | 替换当前硬编码卡片，从 Provider API 拉取 |
| `/blueprints/:id` | 蓝图详情 | 数据服务规格书（SRS）布局——方案概要 → 输入/输出契约 → 处理过程详解 |

### 详情页设计方向：数据服务规格书（SRS）

详情页不是简单的三格式切换，而是面向客户的"服务承诺书"布局，回答三个问题：

1. **输入契约**（客户需要提供什么规格的数据？）
2. **处理逻辑**（我们将对这些数据执行哪些具体的业务操作？）
3. **输出契约**（最终交付的数据规格是什么？）

页面采用自上而下流转布局，三个模块：
- **方案概要**：业务名称、版本号、业务目标描述
- **输入与输出规格**：左右双栏对比，输入侧标约束条件，输出侧标质量承诺
- **处理过程详解**：垂直步骤时间轴，每步包含业务逻辑说明 + 预期产出

详见 [Blueprint 详情页设计方案](docs/blueprint-detail-page-design.md)。

### 数据对接

| 数据 | 来源 | 说明 |
|------|------|------|
| 蓝图列表 | `GET /blueprints` | 蓝图名称、契约摘要、管道名称、验收规则数 |
| 蓝图详情 | `GET /blueprints/:id` | 完整蓝图内容（方案概要 + 输入输出契约 + 处理步骤） |
| 蓝图元数据 | 同上 | 负责人、复核人、版本历史、对应仓库 |

### 交互流程

参考 CLI 五命令工作流，Studio 侧重"看"和"管"：

```
蓝图列表 → 点击进入详情 → 方案概要 → 输入/输出契约对比 → 处理过程步骤流
```

- v0.1.0 不做编辑，先打通"浏览"链路
- 详情页展示业务语言而非代码，客户无需技术背景即可理解
- 后续版本逐步加入 review/design/formalize 操作

### 实现顺序

1. 蓝图列表页：Provider API 对接，替换硬编码数据
2. 蓝图详情页：路由参数 + 三格式 tab 切换
3. 版本历史组件：git-based version list
4. 错误/空态处理：API 不可用时的降级展示

### 依赖

- Provider 需实现 `GET /blueprints` 和 `GET /blueprints/:id` 端点
- Provider 待实现时，Studio 先 mock 数据开发
- 底层 CUE 契约/管道需扩展业务元数据字段（`doc`, `constraint`, `name`, `description`），以支撑详情页的业务化展示

### alpha.1 发布目标

详情页开发完成后发 `studio/v0.1.0-alpha.1`，交付物：
