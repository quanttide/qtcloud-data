## 一、核心设计理念：数据服务规格书（SRS）

将数据蓝图视作一份对客户的"服务承诺书"。页面必须清晰回答三个问题：

1. **输入契约**：客户需要提供什么规格的数据？（责任在客户）
2. **处理逻辑**：我们将对这些数据执行哪些具体的业务操作？（过程透明）
3. **输出契约**：最终交付的数据规格是什么？（承诺结果）

---

## 二、页面结构：自上而下流转逻辑

### 模块一：方案概要 (Blueprint)

- **内容**：方案的业务名称、版本号、一段通俗易懂的业务目标描述
- **目的**：让客户一眼明白这个方案是解决什么问题的

### 模块二：输入与输出规格 (基于 Contract 的双向展示)

数据契约定义了结构约束，拆分为"要求客户提供的"和"承诺交付的"。

**左侧：您需提供的数据（输入契约）**
- 表格展示：字段名、数据类型、业务含义、约束条件（如：必填、格式必须是 YYYY-MM-DD、枚举值为 [M, F]）
- 业务提示：如果客户提供的数据不符合此规格，可能导致处理失败或延迟

**右侧：我们将交付的数据（输出契约）**
- 表格展示：字段名、数据类型、业务含义、质量承诺（如：去重率 100%、空值率 < 1%）
- 业务提示：此规格为最终交付物的验收标准

### 模块三：处理过程详解 (基于 Pipeline 的业务化展开)

不展示 `python3 validate.py`，而是将 `steps` 列表展开为业务逻辑流。

- **展示形式**：垂直的步骤时间轴或卡片流
- **每个步骤包含**：
  - 步骤序号与名称（如"步骤 1：数据格式校验"）
  - 业务逻辑说明（自然语言描述具体做了什么）
  - 预期产出（该步骤完成后数据的中间状态）

## 四、底层工程标准支撑

虽然页面展示的是业务语言，但一切都需要底层的 CUE 定义来支撑。在编写数据契约和管道时，需要预留或扩展业务描述字段。

### 4.1 契约文件支撑交接规格

底层定义需包含输入和输出的 Schema，以及每个字段的约束和业务说明。

例如，在 CUE 中定义字段时：
- `doc: "用户唯一标识"` — 业务含义
- `constraint: "required, non-numeric"` — 约束条件

### 4.2 管道文件支撑处理过程

`steps` 列表中的每一个 step，除了 `command`，还需要有业务元数据。

CUE step 结构可扩展为：
```cue
{
  command: "..."
  name: "数据格式校验"
  description: "检查必填字段..."
}
```

## 五、边界说明

- 只展示蓝图的**设计意图**和**业务逻辑**，不涉及运行时进度
- 客户无需看到代码，就能清楚自己需要提供什么、系统会怎么处理、最终会拿到什么
- 页面实现后，根据实际情况更新本文档

## 六、实现依赖

- 底层 CUE 契约/管道需扩展业务元数据字段（`doc`, `constraint`, `name`, `description`）
- 对应 CLI 的 `preview` 命令从 CUE 生成 HTML 时需读取这些扩展字段
- Studio 展示时直接消费 Provider 返回的结构化数据
